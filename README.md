# Retrieval-Augmented Generation for Question Answering (SQuAD_V2)

This repository demonstrates a **Retrieval-Augmented Generation (RAG) QA system** built on **SQuAD v2**, combining **FAISS-based retrieval** with a **Hugging Face LLM (FLAN-T5)** for answer generation.

This is a continuation of the [qa-bert-squad](https://github.com/Khaghshenas/qa-bert-squad) project, where in the original version, you have a question and a context, and the trained BERT-based model generates an answer directly from that context. In contrast, here in RAG-QA, the system first retrieves relevant context chunks from a larger document collection and then uses a generator (LLM) to produce the answer, which allows it to answer questions using multiple sources.

The **Project Goal** is to:

- Integrate retrieval and generation so answers are grounded in context.
- Use modular design: separate preprocessing, chunking, embeddings, retrieval, and generation.
- Leverage embeddings + FAISS for fast document retrieval.
- Employ an instruction-tuned LLM (flan-t5-large) to generate precise, context-aware answers.

---
## Key Features

- **End-to-End RAG Pipeline**: From document ingestion and chunking to FAISS retrieval and LLM generation.
- **Cross-Encoder Re-Ranking**: Applied using the CrossEncoder from sentence_transformers with the pre-trained ms-marco-MiniLM-L-12-v2 model to rescore the top-k chunks retrieved by FAISS, improving retrieval accuracy before passing the results to the generator.
- **Unique Document Filtering**: Ensures duplicate documents are removed during preprocessing, reducing redundancy and improving retrieval diversity.
- **Explainable Answers**: Retrieved chunks and metadata are returned alongside answers for grounding.
- **Lightweight and Modular**: Can run on CPU or small-to-medium GPUs, and allows easy swapping of embeddings, LLMs, or indexing backends.

## Evaluation
We evaluate the RAG system by separating retrieval quality from end-to-end question answering performance. This is essential to correctly diagnose system behavior. The validation set of the SQuAD v2.0 dataset is used as the evaluation data.

### Retriever Evaluation
**Metric: Recall@K**

Recall@K measures the fraction of questions for which at least one of the top-K retrieved chunks contains the ground-truth answer.

Recall@K = (# questions with answer in top-K retrieved chunks) / (# total questions)

This metric provides a hard upper bound on the achievable end-to-end RAG performance. This means if the answer is not retrieved, the generator cannot answer correctly. 

**Results**

| Metric                | Corpus / Setup     | Value     | Notes                                                 |
| --------------------- | ------------------ | --------- | ----------------------------------------------------- |
| Recall@5 (Oracle)     | Train → Train      | **0.91**  | Upper bound; answer exists in corpus                  |
| Recall@10 (Oracle)    | Train → Train      | **0.92**  | Increasing top-K improves coverage                    |
| Recall@5 (Realistic)  | Train → Validation | **0.19**  | Many validation answers missing from train corpus     |
| Recall@10 (Realistic) | Train → Validation | **0.232** | Higher K helps slightly                               |
| Answer existence rate | Train → Validation | **0.54**  | Upper bound for Recall@∞ (corpus coverage)            |
| Recall@10 efficiency  | Train → Validation | **0.43**  | Fraction of available answers retrieved within top-10 |

**Interpretations**
- When the answer is present in the corpus (Oracle retrieval), the retriever performs very well (Recall@5 = 0.91).
- On realistic questions (Train → Validation), recall drops because many answers are not in the corpus (Recall@5 = 0.19).
- Only about 54% of validation answers exist in the training corpus, so this is the upper limit for retrieval performance.
- Retrieving the top 10 chunks improves coverage, but still only ~43% of available answers are retrieved efficiently.
- In short: the retriever works well when the answer is in the corpus, but end-to-end performance is limited by missing information.

### End-to-End RAG Evaluation
**Metrics: Exact Match (EM), Token-level F1**

Exact Match (EM) measures the percentage of predictions that exactly match any ground-truth answer.

Token-level F1 measures overlap between predicted and ground-truth answer tokens.

**Results**

RAG EM = 0.02 (~2%)
- Only 2% of the generated answers exactly match the ground truth.
- Extremely low, but consistent with the low retrieval recall.

RAG F1 = 0.045 (~4.5%)
- F1 > EM because the generator sometimes produces partial or paraphrased answers, but coverage is still very low.

**Interpretation**
- The RAG system is currently limited by retrieval coverage.
- End-to-end EM/F1 is low because most validation answers are missing from the top-5 retrieved chunks.
- Improving corpus coverage, chunking, or top-K retrieval will likely yield the biggest gains.

## Next Steps
Here’s what we plan to improve next in this RAG system:

1 - Expand corpus coverage: We will add more documents to cover missing answers, especially for validation and potential real-world questions. This will help increase both retrieval recall and end-to-end performance.

2 - Refine document chunking: We will reduce chunk sizes (e.g., 200–300 tokens with overlap) to make sure answers fit fully within single chunks, improving retrieval and generation accuracy.

3 - Improve generator prompting: Currently, we am using a small CPU-friendly LLM, which limits generation speed and possibly quality. We will experiment with other LLMs, including larger or GPU-optimized models, and refine the prompts to boost EM and F1 scores when the answer is present in the retrieved chunks.

4 - Continuous evaluation: We will regularly monitor retrieval Recall@K and end-to-end RAG EM/F1 to measure improvements and guide further optimizations.

Even though the pipeline already includes a re-ranker, the focus will be on improving retrieval coverage, chunk quality, and prompting to maximize the system’s ability to answer questions correctly.

## Setup & Installation

1. Clone the repository:
```bash
git clone https://github.com/yourusername/rag-qa-squad_v2.git
cd rag-qa-squad_v2
```

2. Create a virtual environment and install requirements:

Create the virtual environment (only needed once):
```bash
python -m venv .venv-rag   # Windows
python3 -m venv .venv-rag # Linux / macOS
```
After the environment has been created, activate it:
```bash
.venv-rag\Scripts\Activate.ps1   # Windows (PowerShell)
source .venv-rag/bin/activate # Linux / macOS
```
Once the environment is activated, install the dependencies:
```bash
pip install -r requirements.txt
```

## Step-by-Step Usage

1. Download the Data
```bash
python scripts/download_data.py
```
This downloads the SQuAD V2 dataset to data/raw/squad_v2.

2. Extract Documents
```bash
python scripts/extract_documents.py 
```
This generates documents.jsonl containing each passage/document as a separate entry with metadata.

3. Chunk Documents
```bash
python data/preprocessing/chunk_documents.py
```
This generates chunks.jsonl containing text chunks and metadata (chunk IDs, source document info).

4. Generate Embeddings
```bash
python retriever/embeddings.py
```
This computes embeddings for each chunk using a Hugging Face or Sentence-Transformer model and save them, along with chunk texts, IDs, and metadata, in a .npz file.

5. Build FAISS Index
```bash
python retriever/index.py
```
This builds a FAISS index for fast retrieval

6. Test Generator (LLM)
```bash
python .\scripts\test_generator.py
```
7. Run End-to-End RAG Pipeline
```bash
python .\scripts\test_pipeline.py
```
