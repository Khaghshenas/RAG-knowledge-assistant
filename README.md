# Retrieval-Augmented Generation (RAG) System for Question Answering

This repository implements a **Retrieval-Augmented Generation (RAG) QA system** built on **SQuAD v2** dataset, combining **FAISS-based retrieval** with a pre-trained model, **Hugging Face LLM (FLAN-T5)**, for answer generation.

This is an extension of the [qa-bert-squad](https://github.com/Khaghshenas/qa-bert-squad) project, where in the original version, you have a question and a context, and the trained BERT-based model generates an answer directly from that context. In contrast, here in RAG-QA, the system first retrieves relevant context chunks from a larger document collection and then uses a generator (LLM) to produce the answer, which allows it to answer questions using multiple sources.

In this project we integrate retrieval and generation into a unified pipeline where answers are explicitly grounded in retrieved context. The system follows a modular architecture that separates the preprocessing ETL pipeline - including document extraction, chunking, embedding generation, and vector indexing - from the inference pipeline, which consists of retrieval (searching and reranking) and answer generation, to ensure clarity, scalability, and extensibility. 

## Key Features

- **ETL Pipeline** (Batch Job – Custom Python Pipeline, Procedural Orchestration): Handles document ingestion, preprocessing, and preparation for retrieval.

- **Inference Pipeline** (Service Orchestration – Layered, Stateful Orchestration): Performs retrieval and LLM-based answer generation in a modular, stateful service architecture.

- **Cross-Encoder Re-Ranking**: Uses the CrossEncoder from sentence_transformers with the pre-trained ms-marco-MiniLM-L-12-v2 model to rescore top-k chunks from FAISS, improving retrieval accuracy before generation.

- **Explainable Answers**: Returns retrieved chunks and metadata alongside answers to provide grounding.

- **Lightweight and Modular**: Can run on CPU or small-to-medium GPUs, with easy swapping of embeddings, LLMs, or indexing backends.

## Architecture

The RAG Assistant follows a modular architecture that separates document ingestion, indexing, retrieval, and response generation. This design makes the system easy to extend, evaluate, and deploy.

```text
   ┌─────────────────────────┐
   │ Document Processing     │
   │ • Download dataset      │
   │ (SQuAD v2 dataset)      │
   │ • Context extraction    │
   │ • Chunking              │
   │ • Embedding             │
   │ • FAISS indexing        │
   └────────────┬────────────┘
                │
      Vector Knowledge Base
────────────────┼─────────────────
                │
                ▼
User Question
      │
      ▼
┌──────────────────────┐
│ FAISS Retriever      │
│ • Semantic search    │
│ • Top-K retrieval    │
└───────────┬──────────┘
   Retrieved Candidates
            │
            ▼
┌───────────────────────────┐
│ Cross-Encoder Reranker    │
│ • Relevance scoring       │
│ • Top-K selection         │
└────────────┬──────────────┘
    Relevant Context
             │
             ▼
  ┌───────────────────┐
  │   Prompt Builder  │
  └──────────┬────────┘
             │
             ▼
  ┌───────────────────┐
  │   LLM Generator   │
  └──────────┬────────┘
             │
             ▼
      Final Answer
```
### Components

* **Document Processing & Indexing Pipeline** - Downloads and prepares the SQuAD v2 dataset containing Wikipedia-based contexts, extracts and preprocesses text, splits documents into retrieval-optimized chunks, generates dense embeddings, and builds a FAISS vector index for semantic search.
* **FAISS Retriever** - Retrieves the most relevant document chunks for a user query using vector similarity search and returns top-K candidate contexts.
* **Cross-Encoder Reranker** - Improves retrieval precision by evaluating query-document relevance and reranking the retrieved candidates before generation.
* **Prompt Builder** - Constructs the final prompt by combining the user question with the selected relevant contexts retrieved from the knowledge base.
* **LLM Generator** - Generates the final answer using the retrieved context and user query.

## API Usage

The platform exposes a REST API built with FastAPI for real-time inference. To start the API Service, first run the application locally.

```bash
uvicorn app.api:app --reload --port 8000
```

Once the service is running, open [http://127.0.0.1:8000/docs](http://127.0.0.1:8000/docs) in your browser.

## Local Setup

To get the project running locally, follow these steps in order.

**1. Environment Setup**

First, clone the repo and set up the virtual environment:
```bash
git clone https://github.com/yourusername/rag-qa-squad_v2.git
cd rag-qa-squad_v2
python3 -m venv .venv-rag  # python -m venv .venv-rag for Windows
source .venv-rag/bin/activate # .venv-rag\Scripts\Activate.ps1 for Windows (PowerShell)
pip install -r requirements.txt
```

**2. ETL Pipeline**

You can run the entire ETL pipeline, which downloads the data and generates all the processed files required for the inference pipeline to function.
```bash
python -m src.etl.etl_pipeline
```
**3. Inference Pipeline**

You can use ```run_rag_pipeline``` to run a full RAG workflow on sample data. This executes retrieval and generation steps, and verifies that the pipeline works correctly.
```bash
python -m src.scripts.run_rag_pipeline
```

You can also use ```run_generator``` to run only the LLM-based answer generation on pre-retrieved context. This is useful for validating the model’s output without performing retrieval.
```bash
python -m src.scripts.run_generator
```

## Evaluation
We evaluate the RAG system by separating retrieval quality from end-to-end question answering performance. This is essential to correctly diagnose system behavior. The validation set of the SQuAD v2.0 dataset is used as the evaluation data.

### Retriever Evaluation
**Metric: Recall@K**

Recall@K measures the fraction of questions for which at least one of the top-K retrieved chunks contains the ground-truth answer.

Recall@K = (# questions with answer in top-K retrieved chunks) / (# total questions)

This metric provides a hard upper bound on the achievable end-to-end RAG performance. This means if the answer is not retrieved, the generator cannot answer correctly. 

**Results**

| Metric                | Corpus / Setup     | Value     | Notes                                                 |
| --------------------- | ------------------ | --------- | ----------------------------------------------------- |
| Recall@5 (Oracle)     | Train → Train      | **0.91**  | Upper bound; answer exists in corpus                  |
| Recall@10 (Oracle)    | Train → Train      | **0.92**  | Increasing top-K improves coverage                    |
| Recall@5 (Realistic)  | Train → Validation | **0.12**  | Many validation answers missing from train corpus     |
| Recall@10 (Realistic) | Train → Validation | **0.232** | Higher K helps slightly                               |
| Answer existence rate | Train → Validation | **0.54**  | Upper bound for Recall@∞ (corpus coverage)            |
| Recall@10 efficiency  | Train → Validation | **0.43**  | Fraction of available answers retrieved within top-10 |

**Interpretations**
- When the answer is present in the corpus (Oracle retrieval), the retriever performs very well (Recall@5 = 0.91).
- On realistic questions (Train → Validation), recall drops because many answers are not in the corpus (Recall@5 = 0.12).
- Only about 54% of validation answers exist in the training corpus, so this is the upper limit for retrieval performance.
- Retrieving the top 10 chunks improves coverage, but still only ~43% of available answers are retrieved efficiently.
- In short: the retriever works well when the answer is in the corpus, but end-to-end performance is limited by missing information.

### End-to-End RAG Evaluation
**Metrics: Exact Match (EM), Token-level F1**

Exact Match (EM) measures the percentage of predictions that exactly match any ground-truth answer.

Token-level F1 measures overlap between predicted and ground-truth answer tokens.

**Results**

RAG EM = 0.02 (~2%)
- Only 2% of the generated answers exactly match the ground truth.
- Extremely low, but consistent with the low retrieval recall.

RAG F1 = 0.045 (~4.5%)
- F1 > EM because the generator sometimes produces partial or paraphrased answers, but coverage is still very low.

**Interpretation**
- The RAG system is currently limited by retrieval coverage.
- End-to-end EM/F1 is low because most validation answers are missing from the top-5 retrieved chunks.
- Improving corpus coverage, chunking, or top-K retrieval will likely yield the biggest gains.

## Next Steps
Here’s what we plan to improve next in this RAG system:

1 - Expand corpus coverage: We will add more documents to cover missing answers, especially for validation and potential real-world questions. This will help increase both retrieval recall and end-to-end performance.

2 - Refine document chunking: We will reduce chunk sizes (e.g., 200–300 tokens with overlap) to make sure answers fit fully within single chunks, improving retrieval and generation accuracy.

3 - Improve generator prompting: Currently, we am using a small CPU-friendly LLM, which limits generation speed and possibly quality. We will experiment with other LLMs, including larger or GPU-optimized models, and refine the prompts to boost EM and F1 scores when the answer is present in the retrieved chunks.

4 - Continuous evaluation: We will regularly monitor retrieval Recall@K and end-to-end RAG EM/F1 to measure improvements and guide further optimizations.

Even though the pipeline already includes a re-ranker, the focus will be on improving retrieval coverage, chunk quality, and prompting to maximize the system’s ability to answer questions correctly.
